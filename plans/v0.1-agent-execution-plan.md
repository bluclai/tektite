# Tektite v0.1 Agent Execution Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan phase-by-phase. Complete one phase at a time, verify it, then move on. Do not expand scope beyond the v0.1 release boundary.

**Goal:** Ship a believable Tektite v0.1 release that proves Tektite is a trustworthy local-first markdown workspace with real link semantics, search, backlinks, and safe renames.

**Architecture:** Keep the current split intact: Svelte/SvelteKit owns shell, pane UI, editor presentation, and interaction state; Rust/Tauri owns vault I/O, parsing, indexing, search, rename planning/application, and watch flows. v0.1 should optimize for a strong end-to-end note workflow, not feature breadth.

**Tech Stack:** SvelteKit, Svelte 5 runes, TypeScript, Tauri 2, Rust workspace (`tektite-parser`, `tektite-index`, `tektite-vault`, `tektite-search`), SQLite, CodeMirror 6.

---

## Release boundary

### In scope

The release must make this workflow feel trustworthy:

1. Open a vault.
2. Browse files.
3. Open a markdown note.
4. Edit and save.
5. Search the vault.
6. Follow wiki-links.
7. Inspect backlinks.
8. Rename a note with preview.
9. Continue using the vault without stale state or broken references.

### Explicit cuts

These are out of scope unless already stable and free:

- live preview as a release requirement
- rich external conflict workflows
- advanced split-pane work beyond a stable minimum
- sophisticated workspace restore work
- broad command palette expansion
- in-app AI features
- shell redesign
- cosmetic polish that does not improve trust

### Definition of done for v0.1

Tektite v0.1 is ready when a real user can use the app on a real vault for one focused session and come away with this reaction:

**"I can trust this with my notes."**

---

## Execution rules for the agent

1. Do not reopen product-scope debates while implementing.
2. Do not add net-new feature ideas not named in this plan.
3. Prefer deleting unstable surfaces over stabilizing everything.
4. Every phase must end with verification.
5. If a phase reveals a feature is unstable and non-essential, cut it rather than dragging the release.
6. Commit after each phase with a focused message.
7. Keep README and release-facing language aligned with reality.
