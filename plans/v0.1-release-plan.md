# Plan: Tektite v0.1 Release

> Source: current repo state + `README.md` + `docs/prd-app-v1-reconciled.md` + release-scope review  
> Purpose: define the smallest believable first release, what to cut, and the exact standard for shipping

## Release thesis

Tektite v0.1 is not "the full agent workspace."

It is the first trustworthy version of Tektite's core promise:

**A local-first markdown workspace with real link semantics, search, backlinks, and safe renames.**

The goal of v0.1 is to prove four things:

1. Tektite can open and operate on a real markdown vault.
2. Editing feels calm and reliable.
3. Links, search, and backlinks feel meaningfully better than a plain folder of notes.
4. Renaming notes is safe enough to trust.

If v0.1 proves those four things, the release is successful.

---

## Product boundary

### What v0.1 must be

Tektite v0.1 is a **local-first, desktop markdown workspace** for linked notes.

It is for people who want:

- plain markdown files on disk
- a real file tree
- a strong editor
- search that actually helps
- backlinks that make the graph visible
- wiki-links that behave predictably
- safe renames that do not silently break a vault

### What v0.1 is not

Tektite v0.1 is **not** trying to be:

- a full Obsidian replacement
- an AI app
- a chat interface over notes
- a graph-visualization product
- a multi-surface workspace platform
- a "do everything" knowledge environment

The first release proves substrate, not empire.

---

## Architectural decisions for v0.1

Durable decisions that apply to the release scope:

- **Markdown on disk remains source of truth.** Tektite does not invent a proprietary note format.
- **Local-first remains non-negotiable.** The app must be useful offline and without any hosted dependency.
- **Frontend/backend boundary remains clean.** Svelte owns shell, interaction, and editor presentation. Rust owns filesystem access, parsing, indexing, search, link resolution, rename planning/application, and watch flows.
- **The shell stays fixed.** v0.1 extends the current shell and does not reopen titlebar/activity/sidebar/layout debates.
- **Source mode is the stable editing baseline.** Fancy rendering layers only matter if they do not weaken reliability.
- **Index-backed semantics are part of the product.** Search, backlinks, link resolution, and rename behavior should come from the real backend model, not frontend heuristics.
- **Trust beats breadth.** Every shipped feature must make the product feel more reliable, not just more impressive.

## v0.1 in scope

These are the features that define the first release.

### 1. Vault open flow

The app must support:

- open a local vault folder
- recent vaults
- loading the real vault path into the shell
- initial scan/index population

Why this stays:
Tektite is a workspace, not a demo screen.

### 2. File explorer

The app must support:

- browse folders and markdown files
- open files from the tree
- create file
- create folder
- rename through the safe rename flow
- delete if already reliable

Why this stays:
Without real file operations, the app is not usable as a daily workspace.

### 3. Single-note editing that feels solid

The app must support:

- open note into editor
- type with strong responsiveness
- autosave
- explicit save shortcut
- markdown syntax highlighting
- stable source-mode editing

Why this stays:
The editor is the center of trust.

### 4. Wiki-link semantics

The app must support:

- detect wiki-links
- resolve wiki-links predictably
- follow resolved links
- surface unresolved links clearly
- surface ambiguous links intentionally instead of guessing

Autocomplete is useful, but only part of scope if stable.

Why this stays:
This is a core differentiator, not a nice-to-have.

### 5. Search

The app must support:

- indexed search across the vault
- useful ranked results
- open result into the editor

Why this stays:
Search is part of the durable-context promise.

### 6. Backlinks

The app must support:

- backlinks for the current note
- click from backlink result into source note

Why this stays:
Backlinks make the linked-note model feel real instead of cosmetic.

### 7. Safe rename flow

The app must support:

- rename note/file
- preview affected link rewrites
- apply rename and link updates
- trust that rename does not silently corrupt references

Why this stays:
This is one of the strongest painkiller features in the product.

### 8. Basic external file awareness

The app must support enough watcher behavior to keep the workspace trustworthy:

- tree refresh when the vault changes externally
- reindex on external change where needed
- basic signal to the editor if the open file changed externally

Why this stays:
Not because it is flashy, but because a local-first app cannot feel stale or blind.

## Explicit cuts for v0.1

These are intentionally out of scope for the first release unless they are already done, stable, and free.

### Cut 1: Live preview as a release requirement

Reason:
It is attractive, but it is not the product truth. The product truth is trustworthy markdown substrate.

Rule:
If live preview is unstable, it does not ship in v0.1.

### Cut 2: Rich external conflict workflows

Reason:
Conflict UX can expand endlessly. For v0.1, simple detection and a conservative recovery path are enough.

Rule:
Do not turn conflict handling into a product inside the product.

### Cut 3: Full split-pane ambition

Reason:
Pane systems are complexity magnets. They are only worth shipping if already stable and cheap.

Rule:
Single-pane is enough for v0.1. Basic split-pane may ship only if it does not delay the release.

### Cut 4: Full workspace restore sophistication

Reason:
Useful, but not essential to the first trust loop.

Rule:
A simpler persistence model is acceptable if the app still reopens sanely.

### Cut 5: Fancy command palette modes

Reason:
Default file-jump and a few core commands may be enough. Heading mode and broader command depth are not required to prove the product.

Rule:
Do not hold the release for command palette completeness.

### Cut 6: AI features in the product

Reason:
The positioning already speaks to the agent era. The product does not need embedded AI to prove the thesis.

Rule:
No in-app AI work should block v0.1.

### Cut 7: Additional shell/chrome redesign

Reason:
The shell is good enough. Reopening shell debates is drift.

Rule:
No new persistent chrome, no titlebar rethink, no style detours.

### Cut 8: Platform polishing that does not increase trust

Reason:
Tiny visual refinements can consume release energy without improving daily utility.

Rule:
Prefer boring stability over aesthetic perfection.

## Ship standard

Tektite v0.1 is ready to ship when this exact user flow feels reliable:

1. Open a vault.
2. Browse files in the tree.
3. Open a markdown note.
4. Edit the note.
5. Save the note confidently.
6. Search for another note.
7. Follow a wiki-link.
8. Inspect backlinks.
9. Rename a note with preview.
10. Continue working without broken references or stale state.

If that flow feels strong for a real vault, the app is releasable.

If that flow is shaky, no amount of extra features will save the release.

---

## Release acceptance criteria

### Core trust criteria

- [ ] Vault opens from a real folder without import ceremony.
- [ ] File tree reflects disk state well enough to trust.
- [ ] Opening a note is fast and reliable.
- [ ] Typing in the editor feels responsive.
- [ ] Autosave works and explicit save works.
- [ ] Search returns useful results from the current vault.
- [ ] Wiki-links resolve predictably.
- [ ] Unresolved links are visibly distinct.
- [ ] Ambiguous links are surfaced instead of silently misresolved.
- [ ] Backlinks for the current note are correct enough to trust.
- [ ] Rename preview shows what will change before mutation.
- [ ] Rename apply preserves link integrity.
- [ ] External changes do not leave the app obviously stale or misleading.

### Release hygiene criteria

- [ ] README describes Tektite honestly at the current release scope.
- [ ] First-run flow is understandable.
- [ ] A basic release note can explain what the app does and what is intentionally not in v0.1.
- [ ] A new user can install/run the app and understand the core workflow without reading internal plans.

## Recommended release slices

This is the order that keeps the release moving toward proof instead of breadth.

## Slice 1: Trustworthy open-edit-save loop

Target outcome:
A user can open a vault, open a note, edit, and save without fear.

Includes:

- vault open
- file tree basics
- editor load/save/autosave
- editor status feedback

Do not move past this slice if editing still feels fragile.

---

## Slice 2: Linked-note substrate

Target outcome:
A user can navigate a vault as a knowledge workspace, not just as files.

Includes:

- wiki-link recognition
- link resolution
- unresolved state
- ambiguity surfacing
- backlinks

This is where Tektite starts becoming distinct.

---

## Slice 3: Retrieval loop

Target outcome:
A user can find and reopen context quickly.

Includes:

- search indexing
- ranked results
- open result into editor
- optional lightweight command palette file-jump if already stable

---

## Slice 4: Safe reorganization loop

Target outcome:
A user can rename notes without fear.

Includes:

- rename preview
- rename apply
- link rewrite verification

This slice matters more than preview polish.

---

## Slice 5: Release hardening

Target outcome:
The product can survive first outside use.

Includes:

- stale-state fixes
- external change basics
- README/release-note cleanup
- install/run sanity
- cut anything unstable

---

## What to ship even if it feels boring

Boring but valuable work that should win over exciting extras:

- save correctness
- fast vault open
- path consistency
- good error messages
- reliable link resolution
- index correctness
- rename trust
- clear empty states
- sane first-run experience

These are the features users remember as "solid."

---

## What not to confuse with progress

The following can create false momentum if they happen before the core loop is trustworthy:

- more shell polish
- more surface area in the command palette
- richer preview rendering
- extra pane features
- speculative AI integration
- broader workspace metaphors
- architecture cleanup that does not improve the release path

If a task does not strengthen the open → edit → search → link → backlink → rename loop, it is probably not v0.1 work.

---

## Release positioning

When v0.1 ships, the honest positioning should be simple:

**Tektite is a local-first markdown workspace for linked notes. Open a vault, edit markdown, search instantly, follow wiki-links, inspect backlinks, and rename safely without breaking your knowledge base.**

That is enough for the first release.

It is specific, useful, and believable.

---

## Final rule

v0.1 should optimize for one feeling:

**"I can trust this with my notes."**

Not:

- "This has a lot of features"
- "This has impressive architecture"
- "This hints at a big future"

Trust first.
Breadth later.
